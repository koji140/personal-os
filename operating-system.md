# Operating System

自分がどう考え、どう整理し、どう進めるかを明文化する。
AIや他者との協業時の前提を揃えるための文書。

---

## 基本原則

### 1. まず正本を決める
情報が増えるほど、何が正なのかが重要になる。
まず「何を基準に判断するか」を定める。

### 2. 混ぜない
原文、整理版、証拠、実行物、派生物を混ぜない。
役割が違うものは保存場所も分ける。

### 3. 境界を曖昧にしない
責任の所在や受け渡し地点が曖昧なままだと、実務は止まりやすい。
インターフェースを先に決める。

### 4. 最初から完成形を目指さない
まずは人手で回る最小構成を置く。
その後、改善と自動化を重ねる。

### 5. 判断は説明可能であるべき
結論だけでなく、なぜそうしたのかを後から説明できる状態を目指す。

### 6. 試行錯誤も残す
未解決事項、失敗、引っかかった点も重要な資産。
次の自分やAIのために残す。

## 問題の捉え方

何かを整理するとき、ほぼ常に以下の順序で考える：

```
1. これは何の問題か
2. どこが曖昧なのか
3. 何を正本にするか
4. どの層に分けるべきか
5. 境界はどこか
6. 最小運用は何か
7. どう残せば次に使えるか
```

## AIとの分担

### 人がやること
- 目的設定
- 判断基準の決定
- 正本の決定
- 曖昧さの解消
- 最終判断

### AIがやること
- 構造化
- 文書化
- 選択肢の整理
- 差分反映案の作成
- 実行手順の具体化
- 反復作業の代行

### 分担の原則
AIに「いい感じにやって」は避ける。
影響範囲、必須変更点、禁止事項をなるべく明示する。

## 作業の分類：残件管理 vs 論点管理

作業は2種類に分けて扱う。

- **残件管理**（Todo）：単発の作業・確認・一覧で済む残件
- **論点管理**（Issue / Ticket など）：要件・設計・判断・完了条件を伴うテーマ

次のうち **2つ以上** に当てはまる場合は、そのプロジェクトの論点追跡ツールで管理する。

- 決めることがある
- 関連資料をひも付けたい
- 途中経過やコメント追記が発生しそう
- 完了条件を明確にしたい
- 後で経緯を見返したい

AIは壁打ち中に上記に当てはまる論点が出た場合、論点の追跡（Issue 化など）を提案してよい。

## 力が出る状態 / ストレスを感じる状態

### 力が出る状態
- 基準資料が明確に存在する
- 層が分かれている
- 判断理由が残せる
- 小さく始めて改善できる
- AIとの役割分担が明確

### ストレスを感じやすい状態
- 正本が曖昧
- 責任分界が曖昧
- 具体差分ではなく抽象要約ばかり返ってくる
- 実行者への指示に判断の余地が残りすぎている
- 構造なしに情報が増えていく
