# Operating System

自分がどう考え、どう整理し、どう進めるかを明文化する。
AIや他者との協業時の前提を揃えるための文書。

---

## 基本原則

### 1. まず正本を決める
情報が増えるほど、何が正なのかが重要になる。
まず「何を基準に判断するか」を定める。

### 2. 混ぜない
原文、整理版、証拠、実行物、派生物を混ぜない。
役割が違うものは保存場所も分ける。

### 3. 境界を曖昧にしない
責任の所在や受け渡し地点が曖昧なままだと、実務は止まりやすい。
インターフェースを先に決める。

### 4. 最初から完成形を目指さない
まずは人手で回る最小構成を置く。
その後、改善と自動化を重ねる。

### 5. 判断は説明可能であるべき
結論だけでなく、なぜそうしたのかを後から説明できる状態を目指す。

### 6. 試行錯誤も残す
未解決事項、失敗、引っかかった点も重要な資産。
次の自分やAIのために残す。

## 問題の捉え方

何かを整理するとき、ほぼ常に以下の順序で考える：

```
1. これは何の問題か
2. どこが曖昧なのか
3. 何を正本にするか
4. どの層に分けるべきか
5. 境界はどこか
6. 最小運用は何か
7. どう残せば次に使えるか
```

## AIとの分担

### 人がやること
- 目的設定
- 判断基準の決定
- 正本の決定
- 曖昧さの解消
- 最終判断

### AIがやること
- 構造化
- 文書化
- 選択肢の整理
- 差分反映案の作成
- 実行手順の具体化
- 反復作業の代行

### 分担の原則
AIに「いい感じにやって」は避ける。
影響範囲、必須変更点、禁止事項をなるべく明示する。

### Codexとの作業スタイル

Codexとのやりとりでは、通常作業について毎回確認を取らなくてよい。目的・文脈・作業範囲が明らかな場合は、Codexが妥当な判断をして実行まで進める。

確認なしで進めてよいもの：

- 文書追加・文書更新
- Todoや運用メモの追加
- チャンク、評価質問、レビュー用ファイルの作成
- 既存方針に沿ったスクリプト作成・実行
- Gitの状態確認
- 通常のコミット作成
- 既に設定済みのremoteへのpush

確認を取るもの：

- 既存ファイルの削除や大きな上書き
- 公開範囲が変わる操作
- 秘密情報・個人情報に関わる処理
- 外部サービスへの新規公開、公開リポジトリ作成、共有範囲変更
- コストや課金が発生しそうな実行
- 取り返しのつきにくい破壊的操作

作業後は、何を変更したか、どこに保存したか、次に何をすればよいかを簡潔に報告する。

## 作業の分類：残件管理 vs 論点管理

作業は2種類に分けて扱う。

- **残件管理**（Todo）：単発の作業・確認・一覧で済む残件
- **論点管理**（Issue / Ticket など）：要件・設計・判断・完了条件を伴うテーマ

次のうち **2つ以上** に当てはまる場合は、そのプロジェクトの論点追跡ツールで管理する。

- 決めることがある
- 関連資料をひも付けたい
- 途中経過やコメント追記が発生しそう
- 完了条件を明確にしたい
- 後で経緯を見返したい

AIは壁打ち中に上記に当てはまる論点が出た場合、論点の追跡（Issue 化など）を提案してよい。

## 複数論点会議の進め方

複数テーマが絡む会議（タスク洗い出し・優先順位決め・担当割当が混在する場）では、以下の4ステップで進める。

### 4ステップ

1. **全体出し切り**：批判なしに、アイデア・タスク・懸念をすべて並べる（付箋・リストなど形式は問わない）
2. **分類**：出てきたものを「既存カテゴリに収まるか / 新規か」で振り分ける
3. **時間軸化**：カテゴリ別に「いつ動かすか」を並べ替える（四半期・月次などの軸で）
4. **担当者化**：誰がオーナーで誰が参画するかを、実名・役割レベルで決める

ステップをスキップすると、発散と収束が混ざって議論が止まりやすい。

### 意思決定が進みやすくなる条件

- **2部構成を最初に宣言する**：「まず全体を出す（Part 1）、次に絞る（Part 2）」と最初に共有する。途中で細部に引きずられることを防ぐ。
- **批判なしで出し切ると宣言する**：評価の前に量を出す宣言をしておくと、参加者が安心してアイデアを出せる。
- **ゴールから逆算して優先度を決める**：「◯月に動かすなら、◯月には決めていないと間に合わない」という逆算ロジックを使うと、優先度が自然に定まる。
- **物理的に動かして合意を可視化する**：付箋の移動・並び替えなど、操作が見える形にすると合意が早い。
- **現場担当・決裁者・外部視点をそろえる**：各立場の当事者がそろうことで、漏れへの相互補完と全体視座の保持が起きる。
- **ファシリテーターが「収束」を担う**：発散フェーズが終わったら「ここで絞ります」と明示的に宣言する。自然には収束しない。

## 力が出る状態 / ストレスを感じる状態

### 力が出る状態
- 基準資料が明確に存在する
- 層が分かれている
- 判断理由が残せる
- 小さく始めて改善できる
- AIとの役割分担が明確

### ストレスを感じやすい状態
- 正本が曖昧
- 責任分界が曖昧
- 具体差分ではなく抽象要約ばかり返ってくる
- 実行者への指示に判断の余地が残りすぎている
- 構造なしに情報が増えていく
